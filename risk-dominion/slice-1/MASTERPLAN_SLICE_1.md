### 5.2 Tables

Each table is a `pub struct` annotated with `#[spacetimedb::table(accessor = <snake_name>, public)]`. Column constraints are attributes on the fields (`#[primary_key]`, `#[auto_inc]`, `#[unique]`, `#[index(btree)]`). All four tables are `public` so the client can subscribe to them. Rust field names are `snake_case`; the generated TypeScript bindings expose them as `camelCase` (`territory_id` -> `territoryId`).

```rust
#[spacetimedb::table(accessor = military, public)]
pub struct Military {
    #[primary_key]
    pub territory_id: i32,
    pub owner_id: i32,
    pub troop_count: i32,
}

#[spacetimedb::table(accessor = economic, public)]
pub struct Economic {
    #[primary_key]
    pub territory_id: i32,
    pub owner_id: i32,
    pub capital: i32,
}

#[spacetimedb::table(accessor = players, public)]
pub struct Player {
    #[primary_key]
    pub player_id: i32,
    pub player_name: String,
    pub color: String,
    pub action_points: i32,
    pub last_regen_at: i64,
}

#[spacetimedb::table(accessor = game_state, public)]
pub struct GameState {
    #[primary_key]
    pub key: String,
    pub value: String,
}
```

Access rows through the accessor in `ctx.db`, e.g. `ctx.db.military().territory_id().find(id)`, `ctx.db.players().insert(Player { .. })`, `ctx.db.game_state().key().update(row)`.

### 5.5 Scheduled Table

The action-point regeneration timer is driven by a **scheduled table**, not a `scheduled` reducer attribute. The table names its target reducer; arming it with a repeating `ScheduleAt::Interval` makes SpacetimeDB invoke that reducer on the interval.

```rust
/// Drives `regenerate_action_points` on a fixed interval (deterministic reducer).
#[spacetimedb::table(accessor = regen_timer, scheduled(regenerate_action_points))]
pub struct RegenTimer {
    #[primary_key]
    #[auto_inc]
    pub scheduled_id: u64,
    pub scheduled_at: ScheduleAt,
}
```

`start_game` arms it once (see Section 5.6):

```rust
ctx.db.regen_timer().insert(RegenTimer {
    scheduled_id: 0,
    scheduled_at: ScheduleAt::Interval(
        std::time::Duration::from_secs(ACTION_REGEN_SECONDS).into(),
    ),
});
```

### 5.6 Reducers

All client-callable reducers use the `#[spacetimedb::reducer]` macro and take `ctx: &ReducerContext` as the first parameter. They return `Result<(), String>`: success is `Ok(())`, a validation failure is `Err("message".into())`. Reducers cannot return data to the caller; the client observes the outcome through its table subscriptions (and surfaces any `Err` message via the reducer-call promise rejection). There is no `{ success: true }` JSON wire format.

#### `start_game(ctx: &ReducerContext) -> Result<(), String>`

Idempotent. If `game_state` already has a `status` key, return `Ok(())` immediately. Otherwise insert all seed data (players, game_state, military, economic) and arm the `regen_timer` scheduled table (Section 5.5).

Succeeds with `Ok(())`.

#### `military_attack(ctx: &ReducerContext, territory_id: i32, player_id: i32) -> Result<(), String>`

Validation (return the first failure as `Err(..)`):
1. `game_state.status == "active"` -> else `Err("Game has ended.".into())`
2. player_id is 1 or 2 -> else `Err("Invalid player.".into())`
3. territory_id is 1-12 -> else `Err("Invalid territory.".into())`
4. player.action_points >= 1 -> else `Err("Insufficient action points.".into())`
5. Player owns military in at least one adjacent territory -> else `Err("No adjacent territory controlled.".into())`

On success:
1. Decrement player.action_points by 1.
2. Find adjacent territory where player owns military with highest troop_count. Let `attacker_troops` = that value.
3. Let `defender_troops` = military.troop_count for target territory.
4. If `attacker_troops > defender_troops`:
   - Set military.owner_id = player_id.
   - Set military.troop_count = max(attacker_troops - defender_troops, MIN_TROOPS).
   - Call `dimension_owner_change(ctx, player_id)`.
5. If `attacker_troops <= defender_troops`:
   - Set military.troop_count = max(defender_troops - (attacker_troops / 2), MIN_TROOPS). Integer division.
   - Ownership does not change.

Returns `Ok(())`.

#### `economic_invest(ctx: &ReducerContext, territory_id: i32, player_id: i32) -> Result<(), String>`

Validation (return the first failure as `Err(..)`):
1. `game_state.status == "active"` -> else `Err("Game has ended.".into())`
2. player_id is 1 or 2 -> else `Err("Invalid player.".into())`
3. territory_id is 1-12 -> else `Err("Invalid territory.".into())`
4. player.action_points >= 1 -> else `Err("Insufficient action points.".into())`

On success:
1. Decrement player.action_points by 1.
2. Let `current_owner` = economic.owner_id for target territory.
3. Set economic.capital += ECONOMIC_INVEST_AMOUNT (5).
4. If `player_id != current_owner`:
   - You just added 5 capital to a territory you did not own, so your total now exceeds the previous owner's. Set economic.owner_id = player_id and call `dimension_owner_change(ctx, player_id)`.
5. If `player_id == current_owner`: no ownership change. Capital increased.

Returns `Ok(())`.

### 5.7 Internal Function

#### `dimension_owner_change(ctx: &ReducerContext, new_owner: i32)`

**Not a reducer.** A regular private Rust function called inside the calling reducer's transaction after an ownership flip.

1. Count territories where military.owner_id = new_owner AND economic.owner_id = new_owner.
2. If count >= WIN_UNIFIED_TERRITORIES (3):
   - Set game_state where key='status' to 'ended'.
   - Set game_state where key='winner' to the new owner's player_name.

No return value.

### 5.8 Scheduled Reducer

#### `regenerate_action_points(ctx: &ReducerContext, _timer: RegenTimer)`

Driven by the `regen_timer` scheduled table (Section 5.5), which fires every `ACTION_REGEN_SECONDS` (8 seconds). The target reducer takes the scheduled row as its second argument and uses `#[spacetimedb::reducer]` (the schedule lives on the table, not on a reducer attribute). It makes no HTTP calls.

```rust
#[spacetimedb::reducer]
pub fn regenerate_action_points(ctx: &ReducerContext, _timer: RegenTimer) {
    let ts = now_millis(ctx);
    let players: Vec<Player> = ctx.db.players().iter().collect();
    for mut player in players {
        if player.action_points < MAX_ACTION_POINTS {
            player.action_points += 1;
            player.last_regen_at = ts;
            ctx.db.players().player_id().update(player);
        }
    }
}
```

No return value. Clients receive the updated `players` rows via subscription.

## 8. CLIENT UTILITIES

Generated rows carry camelCase fields (`ownerId`, `territoryId`), so the helpers read them accordingly.

```typescript
// app/client/src/utils/territoryHelpers.ts

import { ADJACENCY, TERRITORY_NAMES } from '../constants';
import type { MilitaryRow, EconomicRow } from '../types';

export function getAdjacentTerritories(territoryId: number): number[] {
  return ADJACENCY[territoryId] || [];
}

export function isAdjacent(territoryId: number, targetId: number): boolean {
  return getAdjacentTerritories(territoryId).includes(targetId);
}

export function countUnifiedTerritories(
  military: readonly MilitaryRow[],
  economic: readonly EconomicRow[],
  playerId: number
): number {
  let count = 0;
  for (const m of military) {
    if (m.ownerId === playerId) {
      const e = economic.find(row => row.territoryId === m.territoryId);
      if (e && e.ownerId === playerId) {
        count++;
      }
    }
  }
  return count;
}

export function getValidMilitaryTargets(
  military: readonly MilitaryRow[],
  playerId: number
): number[] {
  const ownedTerritories = military
    .filter(row => row.ownerId === playerId)
    .map(row => row.territoryId);

  const adjacentToOwned = new Set<number>();
  for (const t of ownedTerritories) {
    for (const adj of getAdjacentTerritories(t)) {
      adjacentToOwned.add(adj);
    }
  }

  return Array.from(adjacentToOwned).filter(t => {
    const m = military.find(row => row.territoryId === t);
    return m && m.ownerId !== playerId;
  });
}

export function getTerritoryName(territoryId: number): string {
  return TERRITORY_NAMES[territoryId] || `Territory ${territoryId}`;
}
```

### 9.2 `useSubscriptions.ts`

Custom hook that manages all SpacetimeDB subscriptions via the `useTable` hook from `spacetimedb/react`.

- Subscribes to all 4 tables through their generated accessors: `useTable(tables.military)`, `useTable(tables.economic)`, `useTable(tables.players)`, `useTable(tables.game_state)`.
- Each `useTable(...)` returns `[rows, isReady]`. The SDK delivers the full table on first load and merges incremental updates automatically (no manual primary-key merge needed).
- Returns `{ military, economic, players, gameState, isReady }`, where `isReady` is the conjunction of every table's ready flag.

```typescript
// app/client/src/hooks/useSubscriptions.ts
import { useTable } from "spacetimedb/react";
import { tables } from "../module_bindings";
import type { MilitaryRow, EconomicRow, PlayerRow, GameStateRow } from "../types";

export function useSubscriptions() {
  const [military, militaryReady] = useTable(tables.military);
  const [economic, economicReady] = useTable(tables.economic);
  const [players, playersReady] = useTable(tables.players);
  const [gameState, gameStateReady] = useTable(tables.game_state);

  const isReady = militaryReady && economicReady && playersReady && gameStateReady;

  return {
    military: military as readonly MilitaryRow[],
    economic: economic as readonly EconomicRow[],
    players: players as readonly PlayerRow[],
    gameState: gameState as readonly GameStateRow[],
    isReady,
  };
}
```

### 9.3 `App.tsx`

Top-level component.

- Reads player_id from the URL parameter: `const playerId = parseInt(new URLSearchParams(window.location.search).get('player') || '1');`
- Calls `useSubscriptions()` to get live table data.
- Binds reducers with the `useReducer` hook from `spacetimedb/react`: `const startGame = useReducer(reducers.startGame);`, `const militaryAttack = useReducer(reducers.militaryAttack);`, `const economicInvest = useReducer(reducers.economicInvest);`. Each returns an async function taking a single named-args object, e.g. `militaryAttack({ territoryId, playerId })`, that resolves when the reducer commits and rejects with the `Err` message on failure.
- Once `isReady` and gameState has no `status` row, calls `startGame()` exactly once (guard with a ref so it is not re-fired) to seed the board. `start_game` is idempotent, so a concurrent second client is harmless.
- Derives state: current player's data (camelCase, e.g. `me.actionPoints`), unified counts per player, game status from the `game_state` rows.
- Manages highlight state for drag-and-drop: a set of highlighted territory IDs.
- Layout (top to bottom, left to right):
  - Top-left: PlayerIndicator
  - Top-right: ActionBar
  - Center: Map (fills remaining space)
  - Bottom: CardHand (fixed bar)
  - Overlay: VictoryScreen (when game ended)
- Passes data as props to all children.

## 10. DRAG-AND-DROP LOGIC

1. Player picks up a card from the hand.
2. `onDragStart` fires:
   - If Military card: compute valid targets via `getValidMilitaryTargets()`. Valid territories are those adjacent to the player's military-owned territories where the player does NOT already own Military.
   - If Economic card: all 12 territories are valid targets.
   - Set highlightedTerritories to the valid target IDs.
3. Valid territories render with a yellow glow. Invalid territories render normally.
4. Player drops the card on a territory.
5. `onDragEnd` fires:
   - If the drop target is a valid territory: call the appropriate bound reducer with a single args object, `{ territoryId, playerId }`.
   - If the drop target is invalid or the card is dropped outside any territory: card returns to hand. No reducer call.
   - Clear highlightedTerritories.
6. Subscription delivers updated state. Map re-renders. Card hand updates (action points decreased).

## 12. GENERATION RULES

1. **All arithmetic is integer arithmetic.** No floats in Rust. Use `/` for truncating division.
2. **SpacetimeDB macros (2.4.1):** tables are `pub struct`s with `#[spacetimedb::table(accessor = name, public)]` and column attributes (`#[primary_key]`, `#[auto_inc]`); client-callable reducers use `#[spacetimedb::reducer]`, take `ctx: &ReducerContext` first, and return `Result<(), String>`; the regeneration timer is a **scheduled table** (`#[spacetimedb::table(accessor = regen_timer, scheduled(regenerate_action_points))]`) armed with `ScheduleAt::Interval(Duration::from_secs(8).into())`. Do not use the obsolete `#[spacetimedb(table)]` / `#[spacetimedb(reducer)]` / `#[spacetimedb(scheduled)]` attribute forms. Use `ctx.timestamp`, never `SystemTime`.
3. **Single `lib.rs`** for all server code with clear section comments as shown in Section 3.1.
4. **Generated bindings:** the client imports `tables`, `reducers`, and `DbConnection` from `module_bindings`, produced by `spacetime generate` after the module compiles. Never hand-edit `module_bindings/`.
5. **Tailwind CSS for all styling.** No custom CSS files. Use utility classes directly in JSX.
6. **TypeScript strict mode.** UI view-models in `types.ts`; row types come from the generated bindings (camelCase fields).
7. **Functional components with hooks.** No class components. Use `spacetimedb/react` hooks (`useTable`, `useReducer`) and `<SpacetimeDBProvider>`.
8. **dnd-kit imports:** `import { DndContext, useDraggable } from '@dnd-kit/core'`. Do not use deprecated react-dnd.
9. **Map rendering:** Use plain SVG elements positioned absolutely. Territories do not need to be real geographic shapes; hexagons per `../AESTHETIC.md` Section 4.
10. **No mock data, no test stubs.** All data comes from SpacetimeDB subscriptions.
11. **Error handling:** reducer calls return promises; `.catch` the rejection (which carries the `Err` message), log it, and display a user-friendly message for common failures (e.g. insufficient action points). Successful outcomes appear via subscription updates, not a return value.
12. **The frontend must compile** with `npm run build` without errors. The server must compile with `cargo build` (or `spacetime build`) without errors.
13. **Visual consistency:** All colors, fonts, spacing, and animations must use the design tokens defined in `../AESTHETIC.md`. Use the Tailwind config from that document. Territory hexagons use the X-split quadrant pattern. Card icons are geometric SVGs. No emojis. No em dashes. No custom CSS files.
