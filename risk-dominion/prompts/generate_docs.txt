**Technical Terms:** SpacetimeDB, Reducer, Procedure, Scheduled Reducer, Scheduled Procedure, Subscription, Module Config, Claude, LLM, Fire-and-Forget, Spectator Mode, Replay Mode.

When defining these technical terms, follow the SpacetimeDB 2.4.1 model: a **reducer** is a transactional server function that validates input and mutates tables but cannot return data to the caller or make HTTP calls (clients observe outcomes through subscriptions); a **procedure** is a server function that can return data to the caller and can make HTTP calls via `ctx.http` (this is how Claude is reached); a **scheduled reducer** runs on a timer for HTTP-free ticks (action regen, cultural spread); a **scheduled procedure** runs on a timer and calls Claude (AI reasoning cycles, Strategist cycle). Do not describe threads, `std::thread::spawn`, `reqwest`, or cross-thread queues; that model is obsolete. The Anthropic API key lives in a private `module_config` table seeded by the `set_config` reducer, not in an environment variable read by module code.

Definitions should be accessible to someone with no prior knowledge of the game. Avoid implementation details — those belong in ARCHITECTURE.md.

The client subscribes to live table updates. AI opponents reason through a multi-agent orchestration pipeline powered by Claude. - **SpacetimeDB Layer:** Explain the table architecture (4 dimension tables, player state, game state, event feed, AI state, reasoning log, strategist log, chat log, AI trust scores, and the private `module_config` table). Distinguish the server function types: **reducers** are client-callable, transactional, and HTTP-free (game actions, chat, `set_config`); **procedures** are client-callable, can return data to the caller, and can make HTTP calls via `ctx.http` (queries, intel, AI cycles); **scheduled reducers** run HTTP-free ticks on a timer (action regen, cultural spread); **scheduled procedures** run timed Claude calls (AI reasoning cycles, Strategist cycle). Note that reducers cannot return data or call HTTP, so anything that talks to Claude or returns results is a procedure. AI action submission is a private Rust function invoked inside the AI cycle procedure's transaction (`ctx.with_tx`). Explain subscriptions (one per table, full rows, client merges updates).
- **AI Orchestration:** Explain the Slice 5 multi-agent model. Commander + 4 specialists per AI. Specialists receive domain-specific game state snapshots and narrow prompts. The reasoning cycle is a scheduled procedure: it snapshots state in one transaction, issues the Claude calls via `ctx.http` (each specialist call followed by a commander synthesis call) with no transaction held open, then commits the resulting actions in a second transaction. Use per-request timeouts via `spacetimedb::http::Timeout` (for example 15s for specialists, 30s for the commander). Commander synthesizes recommendations according to persona priorities. Full deliberation chain logged to ai_reasoning_log and queryable via intel. Do not describe `std::thread::spawn`, JoinHandles, or `reqwest`; that threading model is obsolete.
- **Query System:** Explain the natural language pipeline. Player types question → `query_database` procedure → game state snapshot built in a transaction → prompt constructed with JSON format instructions → Claude reached via `ctx.http` → structured response → summary, data_table, highlighted_territories returned directly to the caller (procedures return data to clients) → ResultsPanel renders, Map highlights. Canned queries use optimized pre-formulated prompts. Autocomplete uses a separate procedure and prompt for context-aware suggestions.
- **Client Architecture:** React component tree. App → Map (Territory × 12), CardHand (ActionCard × N), QueryBar (with autocomplete dropdown), ResultsPanel, IntelPanel, EventTicker, StrategistAlerts, ChatPanel, ActionBar, VictoryScreen. State management via the `spacetimedb/react` hooks (`useTable` for live rows, `useReducer` to call reducers, `useProcedure` to call procedures and receive return values), wrapped in a `<SpacetimeDBProvider>`, plus React useState; generated row and field names are camelCase. Hotkey system via document keydown listener with query bar focus suppression. Spectator mode via ?spectator=true URL parameter (read-only, stats overlay). Replay mode via ?replay=true URL parameter (scrubbable post-game timeline).
- **Data Flow Diagram:** A Mermaid sequence diagram showing: Player drags card → Client calls reducer → SpacetimeDB validates and mutates tables in a transaction → Subscription pushes update to all clients → Client re-renders affected components. AI cycle: Scheduled procedure fires → snapshots state in a transaction → issues specialist Claude calls via ctx.http → Commander synthesizes via a final Claude call → actions written through a private ai_submit_actions function inside a second transaction → Tables updated → Subscription pushes to client.
